**src/components/admin/AdminPatientModal.tsx** — "use client":

Props: `booking` (the selected booking object with _id, patientName, patientEmail, patientPhone, reservationNumber, service, slotDate, slotTime, status), `onClose` (closes modal), `onCancelled` (callback after successful cancel — parent refreshes data)

**Modal structure:**
- Backdrop: fixed overlay, semi-transparent dark background (rgba(0,0,0,0.6)), z-index 40, click backdrop to close
- Modal container: centered, max-width 32rem, dark bg (#1e293b), rounded corners, z-index 50, padding 2rem
- Close button (X) in top right corner

**Patient info section:**
- Patient name as heading (large, white text)
- Email: clickable `mailto:` link (styled as blue/accent link)
- Phone: clickable `tel:` link (styled as blue/accent link)
- Service name
- Date: formatted Hungarian date `new Date(slotDate).toLocaleDateString("hu-HU", { year: "numeric", month: "long", day: "numeric", weekday: "long" })`
- Time: slotTime
- Reservation number
- Status badge: "Visszaigazolva" (green) / "Lemondva" (red/muted)

**Three-dot (⋮) menu:**
- Position: top-right area of the modal (near the close button but separate)
- Button: "⋮" character, styled as a small circular button
- On click: toggles a dropdown with a single option: "Időpont lemondása" (styled with red/warning text color)
- The cancel option is ONLY visible when booking status is "confirmed" AND the appointment is more than 24h away (client-side check — the API enforces server-side too)
- Dropdown uses `position: fixed` to avoid clipping by modal overflow (per RESEARCH.md pitfall 6). Z-index 60 (above modal).
- Click outside dropdown closes it

**Cancel confirmation panel:**
- When "Időpont lemondása" is clicked from the three-dot menu → show confirmation section below patient info (replaces the booking history area temporarily OR overlay within modal)
- Text: "Biztosan lemondja ezt az időpontot?" with patient name and date shown
- Optional reason textarea: placeholder "Indoklás (opcionális)" — if filled, sent in cancellation email
- Two buttons: "Lemondás megerősítése" (red/warning button) and "Mégsem" (cancel/go back)
- On confirm: POST to `/api/admin/booking-cancel` with `{ bookingId: booking._id, reason: reason || undefined }`
- Loading state on the confirm button during API call
- On success: call `onCancelled()` callback and close modal
- On error: show error message from API response

**Booking history section:**
- Below the patient info
- Fetch on modal open: GET `/api/admin/bookings?startDate=2020-01-01&endDate=2030-12-31` — BUT this is too broad. Instead, create a separate inline fetch to get all bookings for this patient's email. Use a GROQ query via a small helper or pass through the bookings API with an additional `email` query param.
- ALTERNATIVE (simpler): Add `email` optional query param to GET /api/admin/bookings — if provided, filter bookings by `patientEmail == $email` instead of date range. Return all bookings for that patient.
- UPDATE AdminDashboard or the modal to fetch: `GET /api/admin/bookings?email={encodeURIComponent(booking.patientEmail)}`
- Display as a simple list: date, time, service name, status badge — sorted by slotDate desc
- Label: "Foglalási előzmények" (Booking history)
- Mark the current booking in the history list (e.g., bold or highlighted)
- If only one booking (the current one), show "Nincs korábbi foglalás." muted text

**src/components/admin/AdminDashboard.tsx** — Update:
- Add state: `selectedBooking` (null or booking object)
- Wire `onBookingClick` from AdminDayPanel and AdminWeekView to set `selectedBooking`
- When `selectedBooking` is not null, render `<AdminPatientModal booking={selectedBooking} onClose={() => setSelectedBooking(null)} onCancelled={handleCancelRefresh} />`
- `handleCancelRefresh`: refetch day bookings and month bookings (to update the calendar dots and day panel after cancel), then set `selectedBooking(null)`

**GET /api/admin/bookings update** (if needed):

- This keeps booking history fetching clean without a separate endpoint
