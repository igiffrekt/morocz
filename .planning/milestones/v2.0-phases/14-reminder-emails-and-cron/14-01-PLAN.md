export async function GET(request: NextRequest) {

  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return new Response("Unauthorized", { status: 401 });
  }
  // ... main logic
}

**Budapest timezone window calculation (no extra packages, use Intl API):**

Create a helper function `appointmentToUtcMs(slotDate: string, slotTime: string): number` that:
1. Parses slotDate ("YYYY-MM-DD") and slotTime ("HH:MM") as components
2. Constructs an ISO string with the correct Budapest offset using `Intl.DateTimeFormat` with `timeZone: 'Europe/Budapest'` and `timeZoneName: 'shortOffset'` to dynamically detect CET (+01:00) or CEST (+02:00)
3. Returns the UTC milliseconds timestamp

The approach:
- Use `new Intl.DateTimeFormat('en-GB', { timeZone: 'Europe/Budapest', timeZoneName: 'longOffset' }).formatToParts(new Date(Date.UTC(year, month-1, day, hour, minute)))` to extract the GMT offset
- Parse the offset string (e.g. "GMT+02:00") to get hours and minutes
- Construct a proper `Date` object: `new Date(Date.UTC(year, month-1, day, hour - offsetHours, minute - offsetMinutes))`
- Return `.getTime()`

**Query strategy (hybrid: broad GROQ filter + precise app-side filtering):**

1. Compute `windowStart = now + 20h` and `windowEnd = now + 28h` as UTC Date objects
2. Derive `dateFrom` and `dateTo` as YYYY-MM-DD strings that could possibly contain bookings in this window (convert windowStart/windowEnd to Budapest date strings — at most 2 consecutive dates)
3. Query Sanity using `getWriteClient().fetch()` (NOT sanityFetch/CDN):

```groq
*[
  _type == "booking"
  && !(_id in path("drafts.**"))
  && status == "confirmed"
  && reminderSent != true
  && slotDate >= $dateFrom
  && slotDate <= $dateTo
]{
  _id, patientName, patientEmail, slotDate, slotTime,
  service->{name}
}
```

4. Filter results in application code: for each booking, call `appointmentToUtcMs(booking.slotDate, booking.slotTime)` and check if it falls within `[windowStart.getTime(), windowEnd.getTime())`

**Grouping by patient email (locked decision: combined email for same-day same-patient):**

const grouped = new Map<string, BookingForReminder[]>();

**Email dispatch loop (per patient group):**

For each patient group:
1. Build appointments array: `appointments.map(b => ({ serviceName: b.service?.name ?? "Vizsgálat", date: formatHungarianDate(b.slotDate), time: b.slotTime }))`
2. For the Hungarian date formatting: create a helper `formatHungarianDate(dateStr: string): string` that converts "2026-02-24" to "2026. február 24." using `Intl.DateTimeFormat('hu-HU', { year: 'numeric', month: 'long', day: 'numeric', timeZone: 'Europe/Budapest' })`.
3. Call `buildReminderEmail({ patientName: group[0].patientName, appointments, clinicPhone: "+36 33 888 8888", clinicAddress: "2500 Esztergom, Simor János u. 36." })` — use the same hardcoded clinic contact as other email routes in the project.
4. Email subject: single appointment → `"Emlékeztető: holnapi időpontja — Mórocz Medical"`, multiple → `"Emlékeztető: holnapi időpontjai — Mórocz Medical"`. Formal Hungarian with accented characters.
5. Call `sendEmail({ to: patientEmail, subject, html })` inside try/catch:
   - On success: patch each booking in the group with `getWriteClient().patch(booking._id).set({ reminderSent: true }).commit()`. Increment `succeeded` counter.
   - On failure: do NOT set reminderSent=true (allows retry next hour per locked decision). Increment `failed` counter. Store error message in `errors` array.
6. Service name mapping: if service name `startsWith("Nőgyógyász")`, map to `"Nőgyógyászati vizsgálat"` — consistent with Phase 12 decision.

**Check isEmailConfigured() first:** If Gmail API credentials are not configured, skip all email sending but still log the run (with 0 attempts). Return 200 with a note.

**Audit logging:**

After processing all groups, insert a row into `cronRunLog`:

**Response:** Return 200 with JSON `{ ok: true, attempted, succeeded, failed }`.

**Error handling:** Wrap the entire main logic in try/catch. If an unexpected error occurs, still try to log to cronRunLog with errorDetails, then return 500.

**Important anti-patterns to avoid (from RESEARCH.md):**

- Do NOT throw on individual email failure — catch per-email, continue processing others
- Do NOT hardcode Budapest UTC offset — use Intl API to handle DST dynamically
